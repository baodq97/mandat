# ADR-0005: Identity mode — agent-user pair where proven, service principal as fallback

## Context

Spike S1 (§11) asked whether the Azure DevOps Users hub accepts the Entra agent-identity service-principal subtype, the instrument D5 and the identity plane (§4.1) build the whole mandate model on. Its kill risk was stated in advance: fall back to a standard service principal per role, keep the architecture, lose sponsor and agent-native audit. The S1 run on 2026-07-15 forced exactly that fallback. This ADR records it before MVP code commits to a path.

The run proved the Entra half works end to end in the dogfood tenant: blueprint created (HTTP 201), blueprint service principal created (201), blueprint client-secret credential issued (200), and one agent identity provisioned (201) reporting `servicePrincipalType: ServiceIdentity`. The break is on the tracker side. Adding that agent identity to the ADO organization's Users hub fails deterministically with error VS403283 from the entitlement API and VS860016 from the identity picker, while a plain service-principal control added to the same hub instantly. No Microsoft documentation claims the agent-identity subtype is entitled to ADO today, ADO's own docs enumerate only the Application and ManagedIdentity principal types, and Entra's provisioning model explains the gap structurally: agent identities provision like service principals, and the Users hub is neither a SCIM nor an app-role surface. The second S1 question resolved positively the same day through the paired agent user (a GA Graph object): it entered the org as a normal directory user with a Basic license and holds work-item assignments. What remains blocked is API authentication as the agent identity, because tokens minted through the blueprint identify a principal type the org cannot resolve.

Round 3, the same day, closed that gap for the tracker's read path. A documented three-step token chain (the blueprint acting for the agent identity, then an agent-identity federated-credential exchange, then a user federated-credential grant) mints a delegated token whose claims name the agent user, not the agent identity. Under that token the agent user authenticated to the ADO API, read a work item (HTTP 200), and posted a comment recorded as created by the agent user (HTTP 200). A kill-criterion probe confirmed the grant is revocable: disabling the agent user made the final token step fail with AADSTS50057 inside the token lifetime, and re-enabling restored it. So the org never has to resolve the ServiceIdentity subtype at all; the delegated path routes around the block through the agent user, an object ADO already accepts. S3 (2026-07-16) then closed the write path: from a clean sandbox holding no standing secret, the agent user cloned over HTTPS with the delegated token as the git credential, pushed a branch, and opened a pull request, all attributed to the agent user (PR `createdBy` and commit author/committer are the agent user as directory facts). Read, comment, clone, push, and PR creation under the agent user are now all proven.

## Decision

- **Two identity modes are viable today; recommend the paired agent user where it is proven.** The design carries both and picks per surface:
  - *Recommended, agent-identity plus a paired agent user per role.* The role's token is minted through the S1 round-3 three-step chain, so writes are attributed to the agent user and the sponsor link and Entra-native audit are IAM properties again, not convention. This restores exactly what the original fallback deferred. Proven today for Azure DevOps end to end — read, comment, clone, push, and PR creation (S1 round 3 and S3); adopt it there.
  - *Fallback, one standard Entra service principal per role*, client-credential, in the tenant the ADO organization is connected to. For any surface where the agent-user chain is not yet proven, chiefly non-ADO surfaces, a plain principal carries the work at the same security floor: hourly tokens, no PAT, no on-disk secret in certificate mode. On Azure DevOps the chain now covers read, comment, and the full git write path, so the fallback is the portable floor there, no longer the write path.
- **Config-not-code, now three values.** `identity_mode: service-principal | agent-user-pair | agent-identity`, written by `mandat init` into `/etc/mandat/config.yaml` (§4.10). `agent-user-pair` is the preferred value for ADO; `service-principal` is the portable fallback; `agent-identity` (the direct subtype straight into the Users hub) stays the still-blocked target-state simplification, kept only as the retest asset. The planes consume whichever principals are configured; they never encode the subtype, so moving between modes is a config flip plus provisioning, never a plane code change.
- **The blueprint stays the single credential holder.** Certificate credential over client secret past the spikes (§7, which already root-owns a blueprint certificate under `/etc/mandat/`), and a managed-identity federated credential in production, so no per-role secret reaches disk in any mode. Every downstream token in the pair mode, all three legs of the chain, descends from that one holder.
- **The mandate invariants hold in both modes.** Remit enforcement stays mechanical (sparse checkout, per-role OS user, server policy, never by prompt); writer ≠ scorer stays an IAM property because distinct principals open and probe a PR, backed by the ADO "creator cannot approve own PR" branch policy (§4.1), demonstrated live on the spike repo where the agent user's approval of its own PR does not count and completion is refused with HTTP 403 until a different user approves; tokens expire hourly and the token itself is the git credential over HTTPS, so no secret reaches disk (§4.1, §7); the no-PAT rule (§7) stands.
- **The direct agent-identity subtype stays the target-state simplification.** The spike blueprint and one agent identity remain provisioned as the retest asset, not torn down. Migration criterion is one falsifiable event: the S1 entitlement call adding the agent identity to the ADO Users hub returns success instead of VS403283. Clearing it would collapse the recommended pair into a single object and drop the extra license. Retest cadence is monthly, or on any ADO release note mentioning agent identities.

### What this still defers, and what it no longer does

- **No longer deferred, sponsor-linked audit across the ADO surface.** S1 round 3 and S3 mint and carry a delegated token that names the agent user, so reads, comments, and the git write path (clone, push, PR creation) all carry the sponsor-linked, Entra-native identity the original fallback had parked. This is the exact loss S1's kill risk named, recovered across the whole Azure DevOps surface, and it moots the round-2 open question of whether the MVP should bother provisioning the pairs: on ADO they are the recommended write path, not an optional attribution add-on.
- **No longer open, the git credential path.** S3 (2026-07-16) exercised git-over-HTTPS carrying the delegated token and PR creation under the agent user from a clean sandbox holding no standing secret: clone, push, and PR-create all succeed with the writes attributed to the agent user. So on Azure DevOps the full write path carries the sponsor-linked, Entra-native identity, not just read and comment. The one honest bound: revoking an already-issued token is TTL-scoped (about one hour), while denial of new credentials is immediate (a disabled agent user refuses the chain at once with AADSTS50057).
- **Caveats to validate in a representative tenant before production.** Three remain open, none a known blocker, none exercisable in the dogfood tenant:
  - *Conditional access against the agent user.* The delegated agent-user token is non-interactive, so a customer baseline that requires MFA or a compliant device on all users could refuse it and drop ADO writes back to the service-principal fallback for that tenant. The dogfood tenant cannot test this: it is not licensed for Conditional Access (no Entra ID P1), the same policy layer the ICP's central-IT tenant enforces and this one never sees. Must be tested in a P1-licensed tenant.
  - *Provisioning privilege.* The pair leaned on Global Administrator plus admin consent (`oauth2PermissionGrant`) and directory-object creation in the S1/S3 runs. Whether a central-IT-delegated admin short of Global Administrator can stand up the full pair is untested, and if not, the recommended pair is harder to provision than the fallback it is preferred over. Establish the least-privilege provisioning path.
  - *Per-seat license and terms of service.* One Basic seat per assignable agent user (cost accepted, §12), and the terms-of-service line on a non-human directory seat (a legal check, not a technical one).

## Consequences

- MVP is unblocked without waiting on Microsoft. The pipeline runs end to end from the first dispatched run, and the §10 MVP was already scoped to client-credential identity mode, so this fixes the credential path without moving the slice boundary. Where read and comment suffice on ADO it runs on the recommended pair; elsewhere on the service-principal fallback.
- The audit downgrade is gone for Azure DevOps. Read, comment, clone, push, and PR-create all carry the sponsor link as a real IAM property (round 3 plus S3), an upgrade over the original fallback's display-name convention. The service-principal fallback now applies only to non-ADO surfaces where the chain is unproven. The security floor is otherwise unchanged: hourly tokens, no PAT, no on-disk secret in certificate mode, remit and writer ≠ scorer intact.
- The "audit downgrade gone for ADO" claim holds for any tenant that permits the non-interactive delegated token. For a tenant whose conditional access refuses it, ADO writes fall back to the service principal and its display-name attribution until the CA path is validated in a P1 tenant. This is the one open condition that can narrow the headline, bounded to conditional-access-enforcing tenants and testable before any such deployment.
- The recommended pair costs one extra Basic license per assignable role, the paired agent user's seat. Against that, the blueprint remains the single credential holder, a managed-identity federated credential in production, so there are no per-role secrets on disk in either mode.
- A live retest obligation still exists for the direct subtype, monthly or on ADO release notes, with a one-line pass condition: the S1 entitlement call succeeds. Clearing it would collapse the pair into a single agent-identity object and drop the extra license, so the blueprint and agent identity keep their small standing footprint as the retest asset. Accepted: cheaper than a re-provisioning project later.
- `mandat doctor`'s S1 check (§4.10) asserts whichever mode is configured, the service-principal path or the three-leg agent-user chain, and flips to asserting the direct entitlement call only after that subtype migrates. Doctor stays the deployed proof of the live mode.
- PRD-0001's Reviewer-identity provisioning and the agent-identity-preview prerequisite still hold. The Reviewer runs as an agent-user pair on the surfaces that support it, so writer ≠ scorer survives with the sponsor link intact, or as a standard service principal on the fallback surfaces.

## Alternatives considered

- **Wait for ADO to accept the agent-identity subtype before shipping MVP.** Blocks the product indefinitely on a Microsoft roadmap with no committed date; VS403283 is a present, reproduced fact, not a transient. S1 round 3 makes the wait doubly unnecessary: the paired agent user already delivers a sponsor-linked, revocable token for read and comment without the subtype being entitled at all. Rejected; the §11 kill risk exists precisely to remove this dependency.
- **Ship plain service principal per role as the only mode.** The original fallback, and still the portable floor. Rejected as the ceiling now that round 3 proves the pair recovers sponsor-linked, Entra-native audit for ADO read and comment; the plain service principal stays the fallback for unproven surfaces, not the whole story.
- **Personal access tokens.** The product's anti-thesis (§7, PRD-0001 Problem): a standing, broadly scoped, unattributable credential is the exact failure Mandat exists to kill. Rejected outright.
- **Impersonate regular user accounts for the agents.** Restores an assignee picker and a "person" in ADO, but breaks the mandate model (no sponsor, no revocable scoped grant, a human license standing in for an agent) and is hygiene the product will not adopt near the ADO terms-of-service line. The paired agent user reaches the same assignee-and-person benefit the sanctioned way: a first-class directory object with a sponsor, a revocable delegated grant (round 3 killed it with AADSTS50057 on demand), and no borrowed human seat. Rejected; the pair is the hygienic substitute.
- **Client secret over certificate for the per-role principals.** Simpler to issue, but leaves a long-lived secret on disk where a certificate need not, against §7's on-disk-secret stance. Rejected for anything beyond spikes; certificate preferred.
